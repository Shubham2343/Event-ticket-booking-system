# Event Ticket Booking System

A full-stack event ticket booking application with seat reservation, countdown timer, and booking confirmation. Built with Node.js + Express + MongoDB (backend) and React (frontend).

---

## Prerequisites

- **Node.js** v18+
- **MongoDB** running locally on `mongodb://localhost:27017` (or update `.env`)
- **npm** v8+

---

## Running the Backend

```bash
cd backend
npm install

# Copy and edit env (defaults work for local MongoDB)
cp .env.example .env

# Seed the database with sample events and seats
npm run seed

# Start the server (port 5000)
npm run dev        # with hot-reload (nodemon)
# or
npm start
```

The API will be available at `http://localhost:5000/api`.

---

## Running the Frontend

```bash
cd frontend
npm install

# Start the React dev server (port 3000)
npm start
```

Open `http://localhost:3000` in your browser. The frontend proxies `/api` requests to `http://localhost:5000` automatically.

## Deploying on Render

The repo includes a [`render.yaml`](render.yaml) Blueprint that provisions both services.

### Prerequisites
- A **MongoDB Atlas** cluster (Render does not host MongoDB). Copy its connection string. Use a **replica set** connection string (Atlas is one by default) to enable transactions — the app also works without one.
- The code pushed to a GitHub/GitLab repo.

### Steps
1. In the Render dashboard: **New → Blueprint**, and select your repo. Render reads `render.yaml` and creates:
   - `ticketflow-backend` — Node web service (`backend/`)
   - `ticketflow-frontend` — static site (`frontend/`)
2. Set the values marked `sync: false` (Render prompts for these):
   - **backend → `MONGODB_URI`**: your Atlas connection string.
   - Deploy once so both services get their public URLs.
3. Wire the two services together (Environment tab → save → redeploy):
   - **backend → `CLIENT_ORIGIN`**: the frontend URL, e.g. `https://ticketflow-frontend.onrender.com`
   - **frontend → `REACT_APP_API_URL`**: the backend URL + `/api`, e.g. `https://ticketflow-backend.onrender.com/api`
4. **Seed the database** (once): from the backend service's **Shell** tab, run `npm run seed`.

`JWT_SECRET` is auto-generated by Render. The frontend's SPA routes (`/events/`, `/my-bookings/`, …) are handled by the rewrite rule in `render.yaml`.

> **Note on the free tier:** Render free web services sleep after inactivity, so the first request after idle can take ~30–60s while the backend wakes.

## Assumptions

1. **One reservation per flow** — a user reserves seats, then must confirm or let it expire before reserving again (no stacking).
2. **No payment flow** — booking is confirmed immediately without payment processing.
3. **Seat pricing** — all seats have equal value; `totalAmount` defaults to 0 (price can be added later).
4. **MongoDB transactions** — multi-document transactions require a replica set or MongoDB Atlas. The backend detects this at runtime: on a replica set it wraps seat updates + the reservation/booking write in a transaction; on a standalone `mongod` it falls back to non-transactional mode. **Double-booking is prevented either way** — the guarantee comes from the atomic conditional `updateMany` (filtered on `status: 'available'` plus a `modifiedCount` check), not from the transaction wrapper. The app therefore runs out of the box against a standard standalone MongoDB install.
5. **No admin panel** — events are seeded via `npm run seed`. New events require direct DB insertion or extending the seed script.

## Design Decisions

### Preventing Double Booking

Double booking is prevented using **MongoDB multi-document transactions** with optimistic locking:

1. Within a single transaction, we first clear any expired reservations for the requested seats.
2. We then read all requested seats and verify every one has `status: 'available'`.
3. We issue a single `updateMany` that only updates seats still matching `status: 'available'` — if concurrent requests slip through step 2 simultaneously, the `modifiedCount` check catches any shortfall and aborts the transaction.
4. Only after all seats are successfully updated do we create the `Reservation` document and commit.

This ensures atomicity — no two users can book the same seat even under concurrent load.

### Reservation Expiry

- Reservations expire after **10 minutes** (configurable via `RESERVATION_EXPIRY_MINUTES`).
- The `expiresAt` field is indexed with `expireAfterSeconds: 0` so MongoDB's TTL index auto-deletes expired reservation documents.
- Before every reservation attempt, stale `reserved` seats (past `reservationExpiry`) are released back to `available` within the same transaction.
- The frontend countdown timer calls `onExpire` when time runs out, clearing the reservation state and refreshing seats.

### Frontend Architecture

- **AuthContext** — global auth state (user, token) with login/register/logout methods; token stored in `localStorage`.
- **Axios interceptor** — automatically injects `Authorization: Bearer <token>` on every request and handles 401s by redirecting to `/login`.
- **SeatGrid** — pure presentational component; receives seats + selected state as props and fires `onSeatToggle` callbacks upward.
- **ReservationTimer** — self-contained countdown component that calls `onExpire` when the countdown hits zero, decoupled from booking logic.
- **EventDetail** — orchestrates the full reservation → booking flow: seat selection → POST `/api/reserve` → countdown → POST `/api/bookings` → redirect to success page.
- Seat statuses auto-refresh every **15 seconds** so users see changes made by others in near real-time.
